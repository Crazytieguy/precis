# Issues

## Known issues

- **Per-group stage value tuning** — the initial stage values (1.0, 0.7, 0.6, etc.) are starting points. Review snapshots across languages and tune per-kind values for better output quality.
- **Duplicate source lines in output** — When two groups/symbols cover overlapping source line ranges, the same line can be emitted twice. Visible in the tock snapshot (Go `const` block line 14 `Format24Hour TimeFormat = iota` appears twice). The formatter should deduplicate lines within the same file.

## Observations

Patterns noticed during fixture inspections. When multiple observations point to the same issue, it's ready to implement.

- **Repo config files take budget without payoff** (mdbook) — `triagebot.toml` section headers (assign, shortcut, bot-pull-requests, etc.) occupy ~10 lines with no value for understanding the codebase. `.github/` and `.circleci/` directories are now deprioritized, but individual repo-management config files (bot configs like `triagebot.toml`, `renovate.json`, `release-please-config.json`) still get normal priority.
- **Repetitive "## License" headings in crate READMEs** (mdbook) — Each of the ~8 workspace crates has a README showing "## License" as a heading. These convey no useful information. More generally, boilerplate markdown sections (License, Contributing, etc.) could be deprioritized.
- **Test/integration crates consume disproportionate budget in workspaces** (toasty) — The integration test suite (ExecLog, LoggingDriver, Test, Setup, Isolate) and its proc macro internals (ThreeValuedBool, BoolExpr, KindVariant, parse helpers) together take ~100 lines (~21% of the 4000-token budget). Meanwhile the core engine module (query compilation pipeline with 66 files) is invisible. Test infrastructure crates could be deprioritized in workspace repos.
- **Central enum bodies crowded out by wrapper struct signatures** (toasty_core, toasty_codegen) — The most architecturally important types (`Expr` with 20+ variants, `Operation` with 8 variants, `Statement`, `Type`, `Value`) are all shown as bare declarations without variant lists. Meanwhile ~30 wrapper structs in `src/stmt/expr_*.rs` each get a signature line that adds almost no information beyond what the enum variants themselves would convey. The scheduler prefers many cheap signature items over fewer but more informative enum body lines. Enum bodies for central types are far more valuable for understanding a codebase than individual wrapper struct signatures. Similarly in toasty_codegen, the `Field` struct (id, attrs, name, ty, set_ident, with_ident), `FieldAttr` struct (key, unique, index, auto, column), and `Model` struct bodies are invisible while expand module method signatures consume budget.
- **Many small files with identical patterns dominate budget** (sps_core, mcphost) — The `install/cask/artifacts/` directory has 23 small files each exporting one `install_<type>()` function, plus a mod.rs with 23 `pub mod` + 22 `pub use self::*` re-exports. Together these consume ~38% of the 2000-token budget. Meanwhile complex files like `install/bottle/exec.rs` and `install/bottle/link.rs` get no interior detail. In mcphost, UI styling helpers (`With*` options in block_renderer.go, `Style*` in enhanced_styles.go, `Display*` in cli.go) take ~40+ lines while `internal/tools/` (the core MCP connection pool and tool management layer — the project's defining feature) is completely absent. The scheduling treats each file/symbol independently, so N trivial items scoring ok individually outcompete fewer important items. Related: `pub use` re-exports are redundant when both the submodule and the re-exporting mod file are shown.
- **ARCHITECTURE.md body content displaced by cheap deprioritized items** (cmdk, toasty_codegen) — ARCHITECTURE.md contains the most valuable information in the repo (design philosophy: compound components + items always in tree, DOM authoritative, discarded approaches with reasoning), but precis only shows headings. Meanwhile, website/ content (deprioritized at 0.15×) still consumes ~23% of output with icon function signatures, boilerplate Next.js README, and demo pages. The issue: cheap signature-level items from deprioritized directories outcompete expensive body-line items from high-value documentation. An Explore agent immediately read ARCHITECTURE.md content and understood the "why" of the design. This compounds with the general "many cheap items beat fewer valuable items" pattern. In toasty_codegen, CONTEXT.md has 135 lines of detailed architecture documentation (purpose, responsibilities, change patterns, code generation principles), but only headings are shown — the Explore agent immediately read the body and gained far richer orientation.
- **Repetitive method signatures within a type consume disproportionate budget** (log, sps, ts-pattern) — In log's `src/kv/value.rs`, `VisitValue` trait has ~15 `visit_*` methods shown individually and `Value` has 8 `from_*` constructors. `RecordBuilder` in `src/lib.rs` shows 15 builder-setter signatures that are entirely predictable from `Record`'s fields. Together these occupy ~43 lines (~23% of output) with near-zero information per line beyond the first 2-3 of each pattern. In sps, `Config` has 12 trivially-patterned directory getters (`bin_dir`, `cellar_dir`, `cask_room_dir`, `cask_store_dir`, `opt_dir`, `taps_dir`, `cache_dir`, `logs_dir`, `tmp_dir`, `state_dir`, `man_base_dir`, `sps_root_marker_path`) that each return a `PathBuf` — the pattern is obvious after 2-3. Similarly `Cache` shows all 7 methods. In ts-pattern, `src/types/Pattern.ts` consumes ~45% of the 2000-token budget with chainable type method signatures — `StringChainable` has 7 methods (startsWith, endsWith, minLength, length, maxLength, includes, regex), `NumberChainable` has 8 (between, lt, gt, lte, gte, int, finite, positive, negative), `BigIntChainable` has 7 similar — all trivially predictable after the first 2-3. The `src/types/` directory overall consumes ~65% of budget with type alias signatures, while runtime matching logic (the recursive descent in helpers.ts) and documentation body content get no interior detail. This is the core "many cheap items beat fewer valuable items" pattern — each method/type signature is cheap, so the scheduler emits all of them before spending on expensive doc/body lines.
- **Pure-documentation directories produce heading-only output** (mdbook_guide_src) — The `guide/src/` directory is entirely markdown documentation (~40 files). Precis output is a detailed table of contents (section headings from every file) with virtually no body content. The Explore agent reads actual content and understands what the documentation says — key workflows, concepts, extension mechanisms. The one place body content appears (README.md), it's CSS from a `<style>` block (lines 4-5) rather than the descriptive prose (lines 20-34). This is the documentation-specific version of "many cheap heading items crowd out valuable body items" — hundreds of headings across dozens of files each cost 1 line, while body content competes poorly. Compounds with the ARCHITECTURE.md and log observations.
- **Large interface/type bodies invisible in component libraries** (vaul) — `DialogProps` (~89 lines, lines 50-139) defines the full public API of the drawer component (~25+ props: `open`, `snapPoints`, `shouldScaleBackground`, `dismissible`, `handleOnly`, `modal`, `direction`, `closeThreshold`, `onDrag`, etc.), but precis shows it as a bare type declaration. Similarly `DrawerContextValue` (~33 fields of shared state) is only shown as an interface declaration. These are the most architecturally important types in the library — an Explore agent read them immediately and understood the full API surface. Meanwhile, 7 browser detection function signatures and 8 DOM helper signatures consume budget with lower information density. This is the TypeScript equivalent of "central enum bodies crowded out by wrapper struct signatures" — another instance of "many cheap signature items beat fewer valuable body items."

## Implementation notes

- **Doc comment detection** is partially migrated to tree-sitter. `Symbol.doc_start_line` is computed from AST sibling navigation, with text heuristic fallback. Handles `///`, `/**` (Rust), `/**` JSDoc (TypeScript), `//`/`/*` (Go, C), `#` (Python). Also handles wrapper nodes (`export_statement`, `decorated_definition`) for exported/decorated symbols. Python docstrings remain text-based heuristics. (Multi-line signature detection was also migrated to tree-sitter — `Symbol.sig_end_line` is computed from the AST body/block child, with text heuristic fallback for nodes without detectable body like type aliases and method signatures.)
- **Overload dedup** — consecutive function symbols with the same name are collapsed (keeping the last). Skipped for Go (`init()` functions).

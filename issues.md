## Known issues

- ~~**Per-group stage value tuning**~~ (tuned: reviewed all snapshots across Rust, Go, Python, TS/JS, C. Main finding: Section body used standard 1/n n-decay, making README/Architecture body lines beyond the first nearly invisible. Now uses gentle linear decay `1+0.1(n-1)` — steeper than types/enums (0.05) since prose is more verbose, but gentle enough to show 5-10 useful body lines. ## Observations

Patterns noticed during fixture inspections. When multiple observations point to the same issue, it's ready to implement.

- ~~**Central enum bodies crowded out by wrapper struct signatures**~~ (fixed: `KindCategory::Enum` with high Body stage value 1.5 and gentle n-decay `1+0.05(n-1)` instead of `1/n`. Enum variant lists now outprioritize repetitive wrapper struct signatures. toasty_core/codegen, thiserror, log, mdbook, otree all show enum variants.)
- ~~**Many small files with identical patterns dominate budget**~~ (partially fixed: directory cohort penalty via `dir_cohort_files`/`dir_cohort_avg_symbols` — when many files in the same directory have groups of the same kind_category with low avg symbols per file, a diminishing-returns penalty is applied: `factor = 1 - (1 - 5/n) * uniformity` where uniformity scales inversely with avg symbols. sps_core artifacts reduced from 23 individual files to mod.rs listing, freeing budget for exec.rs/link.rs. d2ts operators reduced from full class+method+factory per file to names-only for most, freeing budget for README headings, multiset body, core types. mcphost UI styling helpers eliminated, freeing budget for models/session/openai methods. semver functions/ directory eliminated. Partial: peepdb adapters have 4-5 methods each (high avg), so the function-level cohort isn't penalized — only the class declarations are. The method signature repetition requires a more sophisticated approach like cross-file symbol name similarity.) ~~Related: `pub use` re-exports are redundant when both the submodule and the re-exporting mod file are shown~~ (fixed: `pub use` re-exports from child modules now deprioritized via `is_reexport` detection).
- **Pure-documentation directories produce heading-only output** (mdbook_guide_src) — The `guide/src/` directory is entirely markdown documentation (~40 files). Precis output is a detailed table of contents (section headings from every file) with virtually no body content. The Explore agent reads actual content and understands what the documentation says — key workflows, concepts, extension mechanisms. The one place body content appears (README.md), it's CSS from a `<style>` block (lines 4-5) rather than the descriptive prose (lines 20-34). This is the documentation-specific version of "many cheap heading items crowd out valuable body items" — hundreds of headings across dozens of files each cost 1 line, while body content competes poorly. Compounds with the ARCHITECTURE.md and log observations.
- ~~**Package metadata and config files consume budget in small libraries**~~ (fixed: `package.json`, `tsconfig*.json`/`jsconfig*.json`, and `environment*.yaml/yml`/`requirements*.txt` now detected as config files (config_factor=0.2). TOML `[tool.*]` sections marked as config per-group so they're deprioritized within otherwise-useful pyproject.toml. mitt: package.json reduced from 22 content lines to file-path-only, freed budget shows README body text, test structure, and function body through line 60. tomli: deeply nested tool config sections (tox.env_run_base, coverage.run/report, mypy.overrides) eliminated, freed budget shows test imports and tomllib.md body content. xlstm: 3 conda environment YAML files eliminated entirely, freed budget shows mLSTMLayer/sLSTMLayer/CausalConv1d class names and __version__. Broadly improved across 18 fixtures.)
- ~~**Large interface/type bodies invisible in component libraries**~~ (fixed: `KindCategory::Type` Body stage value raised from 0.6 to 1.2 with gentle n-decay `1+0.05(n-1)` instead of `1/n`, mirroring the enum variant fix. `TypeAlias` reclassified from `KindCategory::Constant` to `KindCategory::Type` so TS `type X = { ... }` object types get the same treatment as interfaces. vaul: `WithFadeFromProps`/`WithoutFadeFromProps` show full JSDoc + fields, `DialogProps` shows first prop, `SnapPoint` shows fields. swarm: `Agent` shows all 6 fields, `Response`/`Result` fully visible. bareiron: `EntityDataValue` shows fields. Broadly improved across 50 fixtures — struct/class field definitions consistently shown where previously invisible.) ~~Partial: bareiron `typedef struct {}` declarations still show no field content~~ (fixed: for anonymous `typedef struct/union { ... } Name;`, sig_end now spans the entire definition so the name appears at Names stage instead of showing repeated `typedef struct {` with no identifying information. Named typedefs and typedef enums are excluded — named typedefs already show the name on the opening line, and typedef enum variants are high-value body content.)
- ~~**C typedef struct fields invisible**~~ (bareiron) — fixed: root cause was NOT layout body truncation (body lines were correctly computed). The real issue was that `typedef struct {` signatures are meaningless without the type name (which appears on the closing `} Name;` line), and the Signatures stage replaced useful Names-stage output (`BlockChange …`) for just ~1 extra token per symbol. Fix: `compute_sig_end_line` now sets sig_end to the end of anonymous `typedef struct/union` definitions, making the entire definition the "signature." At tight budgets the scheduler shows Names (type names); at generous budgets it shows the full definition. Typedef enums excluded because their variant lists are high-value body content.
- ~~**C preprocessor defines consume budget in large single-file projects**~~ (sqlite_vec) — fixed in three parts: (1) `_TYPE` defines filtered, (2) name prefix cohort penalty — when ≥5 symbols in a group share a long common name prefix (≥50% of shorter name), the group's value is scaled by `5/cohort_size`. This penalizes repetitive naming patterns like `vec0_column_idx_is_*`/`vec0_column_idx_to_*` (10 functions → 0.5× factor), `sds*` (18 functions → 0.28× factor), `replace*` (range.js). (3) Section documented fix — sections (markdown headings, TOML/JSON/YAML sections) are inherently documentation and shouldn't get the `documented=0.5` penalty. Combined with config_factor reduced from 0.2 to 0.1 to compensate. Result: ARCHITECTURE.md now shows shadow table schemas, idxStr encoding with query plan table, and all sub-section headings. README shows project description. Improved 41 snapshots — section body content (README prose, Architecture descriptions, RFC headings) consistently gained budget freed from repetitive function names and config noise.
- **Unsupported languages make multi-language codebases unintelligible** (soluna) — Soluna is a 2D game framework ("Sokol + Lua = Soluna") with a C rendering engine and a Lua scripting layer. Lua (`.lua`) is not a supported extension, so the entire high-level API (`src/lualib/` with ~12 Lua modules: main.lua, soluna.lua, text.lua, layout.lua) and the service architecture (`src/service/` with render.lua, loader.lua, start.lua, gamepad.lua, log.lua — the ltask-based multi-threaded service layer) are completely invisible. The Explore agent read these immediately and understood the multi-threaded service architecture, Lua↔C binding flow, and the actual user-facing API. Meanwhile, precis output is dominated by ~140 lines of C struct definitions and `#define` macros — the low-level implementation details. The C code alone gives almost no hint that this is a Lua framework at all. Additionally, `luamods.c` (the central module registry listing all ~30 C↔Lua bindings) only shows two function names (`soluna_embed`, `soluna_openlibs`) without the module list that would reveal the framework's API surface. Crypto macros in `lcrypt.c` (GET_UINT32, DES_IP, DES_FP, DES_ROUND, LEFTROTATE — ~8 lines) and platform #defines in `entry.c` (~10 lines) consume budget on implementation details. Compounds with README body content being invisible (the tagline "A framework you can use to make 2D games in Lua with multithreading" would have been the single most useful line).

## Implementation notes

- **Doc comment detection** is partially migrated to tree-sitter. `Symbol.doc_start_line` is computed from AST sibling navigation, with text heuristic fallback. Handles `///`, `/**` (Rust), `/**` JSDoc (TypeScript), `//`/`/*` (Go, C), `#` (Python). Also handles wrapper nodes (`export_statement`, `decorated_definition`) for exported/decorated symbols. Python docstrings remain text-based heuristics. (Multi-line signature detection was also migrated to tree-sitter — `Symbol.sig_end_line` is computed from the AST body/block child, with text heuristic fallback for nodes without detectable body like type aliases and method signatures.)
- **Overload dedup** — consecutive function symbols with the same name are collapsed (keeping the last). Skipped for Go (`init()` functions).

# Issues

## Known issues

- **Per-group stage value tuning** — the initial stage values (1.0, 0.7, 0.6, etc.) are starting points. Review snapshots across languages and tune per-kind values for better output quality.

## Observations

Patterns noticed during fixture inspections. When multiple observations point to the same issue, it's ready to implement.

- **Repo config files take budget without payoff** (mdbook) — `triagebot.toml` section headers (assign, shortcut, bot-pull-requests, etc.) occupy ~10 lines with no value for understanding the codebase. `.github/` and `.circleci/` directories are now deprioritized, but individual repo-management config files (bot configs like `triagebot.toml`, `renovate.json`, `release-please-config.json`) still get normal priority.
- **Repetitive "## License" headings in crate READMEs** (mdbook) — Each of the ~8 workspace crates has a README showing "## License" as a heading. These convey no useful information. More generally, boilerplate markdown sections (License, Contributing, etc.) could be deprioritized.
- **Test/integration/experiment code consumes disproportionate budget** (toasty, xlstm) — The integration test suite (ExecLog, LoggingDriver, Test, Setup, Isolate) and its proc macro internals (ThreeValuedBool, BoolExpr, KindVariant, parse helpers) together take ~100 lines (~21% of the 4000-token budget). Meanwhile the core engine module (query compilation pipeline with 66 files) is invisible. Test infrastructure crates could be deprioritized in workspace repos. In xlstm, the `experiments/` directory (~40% of the 2000-token budget) contains training infrastructure (FormLangDataset, DataGen base class, LR schedulers, metrics, 3 parity YAML configs) that is peripheral to understanding the xLSTM architecture. Meanwhile, core config dataclass fields (xLSTMBlockStackConfig's `slstm_at`, `embedding_dim`, `num_blocks`; mLSTMLayerConfig's `qkv_proj_blocksize`, `proj_factor`) and algorithm details in `backends.py` (parallel stabilized computation with causal mask, recurrent 3-state management) are invisible. Experiments directories in ML repos are analogous to test infrastructure crates — numerous small classes/functions that individually score ok but collectively crowd out core architecture.
- **Central enum bodies crowded out by wrapper struct signatures** (toasty_core, toasty_codegen) — The most architecturally important types (`Expr` with 20+ variants, `Operation` with 8 variants, `Statement`, `Type`, `Value`) are all shown as bare declarations without variant lists. Meanwhile ~30 wrapper structs in `src/stmt/expr_*.rs` each get a signature line that adds almost no information beyond what the enum variants themselves would convey. The scheduler prefers many cheap signature items over fewer but more informative enum body lines. Enum bodies for central types are far more valuable for understanding a codebase than individual wrapper struct signatures. Similarly in toasty_codegen, the `Field` struct (id, attrs, name, ty, set_ident, with_ident), `FieldAttr` struct (key, unique, index, auto, column), and `Model` struct bodies are invisible while expand module method signatures consume budget.
- **Many small files with identical patterns dominate budget** (sps_core, mcphost, d2ts_d2ts, peepdb) — The `install/cask/artifacts/` directory has 23 small files each exporting one `install_<type>()` function, plus a mod.rs with 23 `pub mod` + 22 `pub use self::*` re-exports. Together these consume ~38% of the 2000-token budget. Meanwhile complex files like `install/bottle/exec.rs` and `install/bottle/link.rs` get no interior detail. In mcphost, UI styling helpers (`With*` options in block_renderer.go, `Style*` in enhanced_styles.go, `Display*` in cli.go) take ~40+ lines while `internal/tools/` (the core MCP connection pool and tool management layer — the project's defining feature) is completely absent. In d2ts_d2ts, ~20 operator files each follow an identical class+factory function pattern (e.g., `class FilterOperator extends LinearUnaryOperator` + `export function filter`), consuming most of the 2000-token budget on the pattern repeated ~20 times. Meanwhile, the algorithms that make differential dataflow work (reduce's index reconstruction, join's incremental delta joining, iterate's version dimension extension) get no body content at all. In peepdb, 8 database adapter files (MySQL, PostgreSQL, MariaDB, MongoDB, SQLite, Firebase, MSSQL, Oracle) each implement the same ABC interface with identical method signatures (connect, disconnect, fetch_tables, fetch_data) — consuming ~60+ lines of the 2000-token budget. The base class already shows the interface; after 1-2 implementations the remaining 6-7 are pure redundancy. Meanwhile `core.py` bodies (factory function, orchestration logic, number formatting with scientific notation/isclose()/comma separators) and `config.py` bodies (two-mode encryption with keyring vs PBKDF2HMAC, SQLite/Firebase bypass, 10-minute TTL caching) are invisible. The `peepdb_db` subfolder snapshot (1000 tokens) is even worse — the entire budget is consumed by the 8 adapters. The scheduling treats each file/symbol independently, so N trivial items scoring ok individually outcompete fewer important items. Related: `pub use` re-exports are redundant when both the submodule and the re-exporting mod file are shown.
- **ARCHITECTURE.md body content displaced by cheap deprioritized items** (cmdk, toasty_codegen) — ARCHITECTURE.md contains the most valuable information in the repo (design philosophy: compound components + items always in tree, DOM authoritative, discarded approaches with reasoning), but precis only shows headings. Meanwhile, website/ content (deprioritized at 0.15×) still consumes ~23% of output with icon function signatures, boilerplate Next.js README, and demo pages. The issue: cheap signature-level items from deprioritized directories outcompete expensive body-line items from high-value documentation. An Explore agent immediately read ARCHITECTURE.md content and understood the "why" of the design. This compounds with the general "many cheap items beat fewer valuable items" pattern. In toasty_codegen, CONTEXT.md has 135 lines of detailed architecture documentation (purpose, responsibilities, change patterns, code generation principles), but only headings are shown — the Explore agent immediately read the body and gained far richer orientation.
- **Repetitive method names within a type still consume budget** (log, bareiron, d2ts) — Diminishing returns on Signatures count_factor (first 3 at full value, excess at 15%) prevents large groups from reaching the Signatures stage, so method names appear without full type annotations. However, the method NAMES themselves still consume budget at the Names stage (linear count_factor). In bareiron, 55 packet handler names still appear (~55 lines); in log, 13 `visit_*` and 8 `from_*` names still appear. The Names stage could also benefit from sublinear scaling for very large groups, but this risks losing structural overview.
- **Pure-documentation directories produce heading-only output** (mdbook_guide_src) — The `guide/src/` directory is entirely markdown documentation (~40 files). Precis output is a detailed table of contents (section headings from every file) with virtually no body content. The Explore agent reads actual content and understands what the documentation says — key workflows, concepts, extension mechanisms. The one place body content appears (README.md), it's CSS from a `<style>` block (lines 4-5) rather than the descriptive prose (lines 20-34). This is the documentation-specific version of "many cheap heading items crowd out valuable body items" — hundreds of headings across dozens of files each cost 1 line, while body content competes poorly. Compounds with the ARCHITECTURE.md and log observations.
- **Package metadata and config files consume budget in small libraries** (mitt, tomli, xlstm) — For a 1000-token budget on a single-file library, package.json (~22 lines), tsconfig.json (~6 lines), and README boilerplate headings (License, Contribute, Reporting Issues, Submitting PRs — ~8 lines) together consume ~50% of output. Meanwhile the `mitt()` function body — the actual on/off/emit implementations with key design decisions (`.slice()` copy before iteration, `>>> 0` bitwise trick, wildcard-after-specific ordering) — is truncated at `return {`. The Explore agent read the full 124-line file immediately and understood the implementation. For small libraries, the implementation IS the architecture; type signatures (well-covered by precis) show the "what" but the function bodies show the "how." Compounds with the general "many cheap items beat fewer valuable items" pattern — package.json section headings and README headings are each cheap, collectively outcompeting more expensive body-line items from the single source file. In tomli, `pyproject.toml` `[tool.*]` section headers (isort, tox, coverage, mypy overrides — 18 lines, ~9% of budget) plus supporting scripts (fuzzer, profiler, benchmark — ~13 lines) together consume ~15% of budget with zero architectural value, displacing parser function body content that would show the actual parsing algorithm. In xlstm, 3 conda environment YAML files (`environment_pt220cu121.yaml`, `environment_pt240cu124.yaml`, `environment_pt260cu126.yaml`) consume ~6 lines showing just `name: xlstm / channels: / dependencies:` — zero architectural value.
- **Large interface/type bodies invisible in component libraries** (vaul, swarm, bareiron) — `DialogProps` (~89 lines, lines 50-139) defines the full public API of the drawer component (~25+ props: `open`, `snapPoints`, `shouldScaleBackground`, `dismissible`, `handleOnly`, `modal`, `direction`, `closeThreshold`, `onDrag`, etc.), but precis shows it as a bare type declaration. Similarly `DrawerContextValue` (~33 fields of shared state) is only shown as an interface declaration. These are the most architecturally important types in the library — an Explore agent read them immediately and understood the full API surface. Meanwhile, 7 browser detection function signatures and 8 DOM helper signatures consume budget with lower information density. In swarm, the `Agent(BaseModel)` class has 6 fields (name, model, instructions, functions, tool_choice, parallel_tool_calls) defining the entire framework's API, but precis only shows `name: str = "Agent"` then `…`. `Response` and `Result` model bodies are similarly invisible. Meanwhile, ~100 lines of example function signatures (deprioritized at 0.15× but numerous) crowd out these ~15 lines of core type definitions. This is the TypeScript/Python equivalent of "central enum bodies crowded out by wrapper struct signatures" — another instance of "many cheap signature items beat fewer valuable body items." In bareiron, `globals.h` has 4 `typedef struct {` declarations (PlayerData with 41-slot inventory/uuid/health/hunger/saturation/bit-packed flags, MobData with health/type/position/bit-packed data, BlockChange, ChunkAnchor) — precis shows just `typedef struct {` for each with no struct name or field details. These are the most architecturally important types in the codebase (the Explore agent found PlayerData fields immediately).
- **Auto-generated API doc READMEs duplicate source signatures** (py3xui) — Six subdirectories (api/, async_api/, client/, inbound/, server/, utils/) each contain a pdoc-generated README.md with class/method heading listings that are 100% redundant with the .py file signatures already shown. These consume ~40 lines (~16% of the 2000-token budget) with zero additional information. The Explore agent ignored them entirely. Additionally, the async_api/ module is a direct mirror of api/ (6 files with identical structure, just async/await keywords added), consuming ~30 lines of redundant signatures — compounding with the "many small files with identical patterns" observation. Property getters (host, username, password, use_tls_verify, custom_certificate_path, max_retries, session, cookie_name) are shown for both sync and async base classes — compounding with "repetitive method signatures." Meanwhile, Client model fields (16 fields with snake_case→camelCase aliases, flexible id type), BaseApi._request_with_retry internals (retry with exponential backoff, three-tier TLS handling), JsonStringModel base class, and Inbound.to_json() serialization are all invisible — compounding with "large interface/type bodies invisible."
- **C header guard `#define` lines waste budget** (bareiron) — Six header files each show their include guard macro (`#define CRAFTING_H`, `#define H_PACKETS`, `#define H_PROCEDURES`, `#define H_STRUCTURES`, `#define H_TOOLS`, `#define H_WORLDGEN`) as the only or primary content from those files. These are boilerplate with zero informational value — analogous to package metadata noise but specific to C codebases.
- **C preprocessor defines and typedefs consume budget in large single-file projects** (sqlite_vec) — `sqlite-vec.c` (~9900 lines) has ~20 lines of platform-compatibility preprocessor defines (`#define UINT32_TYPE uint32_t`, `#define UINT16_TYPE unsigned short int`, etc.) and simple typedefs (`typedef int8_t i8; typedef uint8_t u8;` etc.) at the top of the file, consuming ~10% of the 2000-token budget with zero architectural value. These are C boilerplate for type portability — trivially predictable and uninteresting. Meanwhile, VectorElementType (3 values: FLOAT32, INT8, BIT) and Vec0DistanceMetrics (3 values: L2, COSINE, L1) enum bodies are invisible, ARCHITECTURE.md body (explaining vec0 shadow tables, idxStr encoding, chunked storage design) is heading-only, and the KNN search algorithm (chunk-based multi-pass with merge_sorted_lists) gets no interior detail. Additionally, 10 `vec0_column_*_idx*` helper functions follow an obvious pattern (distance_idx, k_idx, is_vector, to_vector_idx, is_partition, to_partition_idx, is_auxiliary, to_auxiliary_idx, is_metadata, to_metadata_idx). Compounds with "ARCHITECTURE.md body content displaced," "repetitive method signatures," and the general "many cheap items beat fewer valuable items" pattern.
- **TypeScript declaration files duplicate JavaScript source signatures** (commander) — `typings/index.d.ts` consumes ~38 lines (~19% of the 2000-token budget) showing typed method signatures for Command, Option, Argument, and Help — all of which are already shown from the `.js` source files. The Explore agent ignored the `.d.ts` entirely. Meanwhile, `suggestSimilar.js` (Damerau-Levenshtein edit distance for "did you mean?" suggestions) is completely absent, and README body content is not shown. In JavaScript projects that ship separate `.d.ts` files alongside `.js` source, the declaration files are largely redundant — the `.js` signatures already convey the API surface, and the type annotations add minimal architectural insight. This compounds with "many cheap items beat fewer valuable items" — each typed signature line is cheap, collectively outcompeting more expensive body content from core methods like `parseOptions()` and `_parseCommand()`.
- **Unsupported languages make multi-language codebases unintelligible** (soluna) — Soluna is a 2D game framework ("Sokol + Lua = Soluna") with a C rendering engine and a Lua scripting layer. Lua (`.lua`) is not a supported extension, so the entire high-level API (`src/lualib/` with ~12 Lua modules: main.lua, soluna.lua, text.lua, layout.lua) and the service architecture (`src/service/` with render.lua, loader.lua, start.lua, gamepad.lua, log.lua — the ltask-based multi-threaded service layer) are completely invisible. The Explore agent read these immediately and understood the multi-threaded service architecture, Lua↔C binding flow, and the actual user-facing API. Meanwhile, precis output is dominated by ~140 lines of C struct definitions and `#define` macros — the low-level implementation details. The C code alone gives almost no hint that this is a Lua framework at all. Additionally, `luamods.c` (the central module registry listing all ~30 C↔Lua bindings) only shows two function names (`soluna_embed`, `soluna_openlibs`) without the module list that would reveal the framework's API surface. Crypto macros in `lcrypt.c` (GET_UINT32, DES_IP, DES_FP, DES_ROUND, LEFTROTATE — ~8 lines) and platform #defines in `entry.c` (~10 lines) consume budget on implementation details. The `web/config.toml` (Zola site config — 2 lines) provides no architectural value. Compounds with README body content being invisible (the tagline "A framework you can use to make 2D games in Lua with multithreading" would have been the single most useful line).

## Implementation notes

- **Doc comment detection** is partially migrated to tree-sitter. `Symbol.doc_start_line` is computed from AST sibling navigation, with text heuristic fallback. Handles `///`, `/**` (Rust), `/**` JSDoc (TypeScript), `//`/`/*` (Go, C), `#` (Python). Also handles wrapper nodes (`export_statement`, `decorated_definition`) for exported/decorated symbols. Python docstrings remain text-based heuristics. (Multi-line signature detection was also migrated to tree-sitter — `Symbol.sig_end_line` is computed from the AST body/block child, with text heuristic fallback for nodes without detectable body like type aliases and method signatures.)
- **Overload dedup** — consecutive function symbols with the same name are collapsed (keeping the last). Skipped for Go (`init()` functions).

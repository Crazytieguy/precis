## Known bugs

- **Cost accounting mismatch** — ~~the scheduling algorithm's cost tracking diverges from actual rendered word counts. Some entries significantly exceed their budget (e.g., 418% at budget 500 for pluggy).~~ The major source of cost divergence was a **double-deduction bug**: when a high-level item (e.g., Doc(3)) was included before a lower-level item (e.g., Names), the high-level item paid for Names as a prerequisite, but the original Names heap entry retained a valid generation number. Popping that stale entry deducted its cost again, wasting budget. Two fixes: (a) skip queue items whose stage is already covered by the group's current included stage, (b) when computing prerequisites for a later stage, correctly handle partially-included prerequisite stages (e.g., when included at Doc(2), don't skip all Doc lines — only skip Doc(1) and Doc(2)). Budget utilization improved from mean 49% to 86%. ~~Remaining small divergences (1-5%) come from the renderer's `emitted_up_to` dedup skipping content that was charged for, particularly when parent symbol bodies overlap with nested symbol ranges.~~ The largest remaining source of overshoot was **unaccounted truncation marker costs**: the renderer appends ` …` (1 word) to each name-only symbol, but the scheduler's `name_words` cost didn't include it. For codebases with many symbols at Names-only level (e.g., commander/lib with ~100 methods), this caused 10-14% budget overshoot. Fixed by including the ellipsis word in `name_words`; the +1 is offset by a -1 in `signature_words` (since `signature_words = sig_total - name_words`), so the total Names+Signatures cost stays exact. Remaining minor overshoots (1-4%) come from standalone `→…` truncation markers on partial doc/body and from `emitted_up_to` dedup.

## Remaining work

- **Residual cost mismatch from emitted_up_to** — the renderer skips symbols whose start line falls within already-emitted ranges (e.g., nested symbols inside trait/impl/function bodies). The scheduler charges for these symbols but they don't render. This causes small overestimation (1-5%). A proper fix would either exclude overlapping body lines from parent symbol costs or teach the scheduler about line-level deduplication.
- ~~**Truncation markers** — `…` inline for truncated lines (e.g. name only instead of full signature), standalone `…` line for omitted content (e.g. remaining doc lines, body). Makes it visually clear what's summarized vs complete. The old rendering system had this; the new one doesn't yet.~~ Implemented: inline ` …` on name-only lines, standalone `→…` after partial doc/body. Body truncation markers are suppressed when truncated lines overlap with nested symbols (e.g., class bodies containing methods rendered individually). The inline ` …` cost is now accounted for in the scheduler. ~~The standalone `→…` markers (1 word each, per-symbol when doc/body is truncated) are still unaccounted — their cost is conditional on whether the group advances to more lines, making it hard to model in the per-stage cost system.~~ Standalone `→…` markers are now accounted for using marker count deltas in `stage_cost`: at Doc/Body(n), the cost includes `markers_at(n) - markers_at(n-1)`, where `markers_at(n) = symbols with > n lines`. The telescoping sum across prerequisites ensures accurate totals. Body markers use a precomputed `body_has_nested` flag to skip symbols where the renderer suppresses the marker. This eliminated all budget overshoots (previously up to 4.6% on commander/lib at budget 2000). Mean utilization decreased slightly (88.9% → 88.2%) because some charged markers correspond to symbols later skipped by `emitted_up_to` dedup.
- **File paths as stages** — currently file paths are shown automatically when any symbol content is included. Making file paths an explicit stage would allow showing just file paths for low-value groups at tight budgets, and would create cheap items that improve budget utilization by filling gaps at the end of scheduling.
- **Import statement rendering** — show imports as a new group kind with its own stage progression. Distinguish 1st-party imports (relative paths, `crate::`, `super::`, Go module path) from 3rd-party. 1st-party imports are high signal for understanding a file's role.
- ~~**README.md priority boost** — render `README.md` at higher priority than other markdown files. Previously attempted with the old level-based system and was utilization-neutral. Should be revisited now that value is per-group.~~ Implemented via `FileRole` grouping dimension: README files get 1.5x value boost, changelog files (CHANGELOG.md, CHANGES.md, HISTORY.md, NEWS.md) get 0.1x penalty. This separates README/changelog/other markdown into distinct groups, preventing changelog headings from consuming budget that should go to README content and code structure.
- **Test/vendor/example directories** — instead of excluding entirely during file discovery, render at lower priority. They appear as paths at tight budgets, content at generous budgets.
- **Per-group stage value tuning** — the initial stage values (1.0, 0.7, 0.6, etc.) are starting points. Review snapshots across languages and tune per-kind values for better output quality.
- **Config file support** — support json/yaml/toml files with different rendering heuristics than code.

## Implementation notes

- **Doc comment and multi-line signature detection** are text-based heuristics. Doc comments handle `///`, `//!`, `/** */`, Go `//` (godoc, language-gated), Python `#` and docstrings. Multi-line signatures scan for `{`/`;`/`:` delimiters. Worth re-evaluating tree-sitter for these as more languages are added.
- **Overload dedup** — consecutive function symbols with the same name are collapsed (keeping the last). Skipped for Go (`init()` functions).

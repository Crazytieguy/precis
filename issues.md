# Issues

## Known issues

- **Per-group stage value tuning** ‚Äî the initial stage values (1.0, 0.7, 0.6, etc.) are starting points. Review snapshots across languages and tune per-kind values for better output quality.
- **Duplicate source lines in output** ‚Äî When two groups/symbols cover overlapping source line ranges, the same line can be emitted twice. Visible in the tock snapshot (Go `const` block line 14 `Format24Hour TimeFormat = iota` appears twice). The formatter should deduplicate lines within the same file.

## Observations

Patterns noticed during fixture inspections. When multiple observations point to the same issue, it's ready to implement.

- **Repo config files take budget without payoff** (mdbook) ‚Äî `triagebot.toml` section headers (assign, shortcut, bot-pull-requests, etc.) occupy ~10 lines with no value for understanding the codebase. `.github/` and `.circleci/` directories are now deprioritized, but individual repo-management config files (bot configs like `triagebot.toml`, `renovate.json`, `release-please-config.json`) still get normal priority.
- **Repetitive "## License" headings in crate READMEs** (mdbook) ‚Äî Each of the ~8 workspace crates has a README showing "## License" as a heading. These convey no useful information. More generally, boilerplate markdown sections (License, Contributing, etc.) could be deprioritized.
- **Test/integration/experiment code consumes disproportionate budget** (toasty, xlstm) ‚Äî The integration test suite (ExecLog, LoggingDriver, Test, Setup, Isolate) and its proc macro internals (ThreeValuedBool, BoolExpr, KindVariant, parse helpers) together take ~100 lines (~21% of the 4000-token budget). Meanwhile the core engine module (query compilation pipeline with 66 files) is invisible. Test infrastructure crates could be deprioritized in workspace repos. In xlstm, the `experiments/` directory (~40% of the 2000-token budget) contains training infrastructure (FormLangDataset, DataGen base class, LR schedulers, metrics, 3 parity YAML configs) that is peripheral to understanding the xLSTM architecture. Meanwhile, core config dataclass fields (xLSTMBlockStackConfig's `slstm_at`, `embedding_dim`, `num_blocks`; mLSTMLayerConfig's `qkv_proj_blocksize`, `proj_factor`) and algorithm details in `backends.py` (parallel stabilized computation with causal mask, recurrent 3-state management) are invisible. Experiments directories in ML repos are analogous to test infrastructure crates ‚Äî numerous small classes/functions that individually score ok but collectively crowd out core architecture.
- **Central enum bodies crowded out by wrapper struct signatures** (toasty_core, toasty_codegen) ‚Äî The most architecturally important types (`Expr` with 20+ variants, `Operation` with 8 variants, `Statement`, `Type`, `Value`) are all shown as bare declarations without variant lists. Meanwhile ~30 wrapper structs in `src/stmt/expr_*.rs` each get a signature line that adds almost no information beyond what the enum variants themselves would convey. The scheduler prefers many cheap signature items over fewer but more informative enum body lines. Enum bodies for central types are far more valuable for understanding a codebase than individual wrapper struct signatures. Similarly in toasty_codegen, the `Field` struct (id, attrs, name, ty, set_ident, with_ident), `FieldAttr` struct (key, unique, index, auto, column), and `Model` struct bodies are invisible while expand module method signatures consume budget.
- **Many small files with identical patterns dominate budget** (sps_core, mcphost, d2ts_d2ts, peepdb) ‚Äî The `install/cask/artifacts/` directory has 23 small files each exporting one `install_<type>()` function, plus a mod.rs with 23 `pub mod` + 22 `pub use self::*` re-exports. Together these consume ~38% of the 2000-token budget. Meanwhile complex files like `install/bottle/exec.rs` and `install/bottle/link.rs` get no interior detail. In mcphost, UI styling helpers (`With*` options in block_renderer.go, `Style*` in enhanced_styles.go, `Display*` in cli.go) take ~40+ lines while `internal/tools/` (the core MCP connection pool and tool management layer ‚Äî the project's defining feature) is completely absent. In d2ts_d2ts, ~20 operator files each follow an identical class+factory function pattern (e.g., `class FilterOperator extends LinearUnaryOperator` + `export function filter`), consuming most of the 2000-token budget on the pattern repeated ~20 times. Meanwhile, the algorithms that make differential dataflow work (reduce's index reconstruction, join's incremental delta joining, iterate's version dimension extension) get no body content at all. In peepdb, 8 database adapter files (MySQL, PostgreSQL, MariaDB, MongoDB, SQLite, Firebase, MSSQL, Oracle) each implement the same ABC interface with identical method signatures (connect, disconnect, fetch_tables, fetch_data) ‚Äî consuming ~60+ lines of the 2000-token budget. The base class already shows the interface; after 1-2 implementations the remaining 6-7 are pure redundancy. Meanwhile `core.py` bodies (factory function, orchestration logic, number formatting with scientific notation/isclose()/comma separators) and `config.py` bodies (two-mode encryption with keyring vs PBKDF2HMAC, SQLite/Firebase bypass, 10-minute TTL caching) are invisible. The `peepdb_db` subfolder snapshot (1000 tokens) is even worse ‚Äî the entire budget is consumed by the 8 adapters. The scheduling treats each file/symbol independently, so N trivial items scoring ok individually outcompete fewer important items. Related: `pub use` re-exports are redundant when both the submodule and the re-exporting mod file are shown.
- **ARCHITECTURE.md body content displaced by cheap deprioritized items** (cmdk, toasty_codegen) ‚Äî ARCHITECTURE.md contains the most valuable information in the repo (design philosophy: compound components + items always in tree, DOM authoritative, discarded approaches with reasoning), but precis only shows headings. The issue: cheap signature-level items from deprioritized directories outcompete expensive body-line items from high-value documentation. ## Implementation notes

- **Doc comment detection** is partially migrated to tree-sitter. `Symbol.doc_start_line` is computed from AST sibling navigation, with text heuristic fallback. Handles `///`, `/**` (Rust), `/**` JSDoc (TypeScript), `//`/`/*` (Go, C), `#` (Python). Also handles wrapper nodes (`export_statement`, `decorated_definition`) for exported/decorated symbols. Python docstrings remain text-based heuristics. (Multi-line signature detection was also migrated to tree-sitter ‚Äî `Symbol.sig_end_line` is computed from the AST body/block child, with text heuristic fallback for nodes without detectable body like type aliases and method signatures.)
- **Overload dedup** ‚Äî consecutive function symbols with the same name are collapsed (keeping the last). Skipped for Go (`init()` functions).
